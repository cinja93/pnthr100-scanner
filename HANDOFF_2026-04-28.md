```markdown
---
name: Earnings-week stops are manually tightened in TWS — TWS is canonical when tighter
description: During earnings weeks for held positions, Scott manually tightens stops in TWS to limit risk through the print. The TWS stop is his discretionary override and PNTHR must defer to it. Rule: when reconciling, the TIGHTEST stop wins (highest for LONG, lowest for SHORT) — Phase 4 must auto-adopt manual tightenings rather than fight them.
type: feedback
---

**RULE:** When IBKR shows a stop tighter than PNTHR's algorithmic PNTHR Stop, treat IBKR as canonical. PNTHR adopts TWS's value. Do NOT push PNTHR's looser PNTHR Stop back to IBKR.

**Why:** Scott manually tightens stops in TWS during earnings weeks (and any other time he's actively managing risk through a known event). The PNTHR Stop algorithm (Wilder ATR + 2-week structural floor) doesn't know the calendar — it doesn't see earnings dates and would suggest a wider stop. Scott's discretionary tightening is the actual risk control. If PNTHR fought it (e.g., pushed a wider stop back to IBKR), it would defeat the purpose of his manual override and could expose him to a bad print.

**How to apply:**
1. **Reconciliation/audit logic:** when comparing IBKR stop to PNTHR stop, if IBKR is tighter, **silently adopt IBKR into PNTHR**. Do NOT raise this as a mismatch needing user attention. Update `pnthr_portfolio.stopPrice` and `stopHistory[]` to log "USER_TIGHTENED_VIA_TWS".
2. **Phase 4 push direction:** PNTHR's algorithmic ratchet only pushes to IBKR when PNTHR's new value is TIGHTER than IBKR's current stop. Otherwise the bridge does nothing — preserves user's manual override.
3. **Tightest stop wins** as the universal rule:
   - LONG: tightest = highest (closest to current price below)
   - SHORT: tightest = lowest (closest to current price above)
4. **Phase 4 detection cron:** every sync, check IBKR stops vs PNTHR stops. If IBKR tighter, adopt. If PNTHR tighter (e.g., after weekly ATR ratchet), push to IBKR.
5. **Discrepancy banner / LIVE table:** the rule "PNTHR is canonical" is OBSOLETE. Replace with "tightest is canonical, both sides converge". A "STOP MISMATCH" alert should ONLY trigger when the two sides differ AND can't be reconciled by adopting the tighter value (rare edge case, e.g. partial fills).

**Examples confirmed 2026-04-28:**
- MU TWS $488 vs PNTHR $430.09 → PNTHR adopts $488 (TWS tighter)
- TWLO $139 vs $125.70, CVNA $394 vs $363.37, NVDA $207 vs $193.58, FIVE $228 vs $215.72, SWK $76 vs $72.97 — all earnings-week tightenings; all should adopt TWS values silently.

1. **Order routing:** SMART (current default).
2. **Stop order type — RTH:** **Pure STP (stop market)** for regular trading hours. NO STP LMT during RTH. Reason: Scott has been missing fills with limit-on-trigger orders during fast moves. Stop market guarantees a fill, accepts whatever price the market gives.
3. **Stop order type — Extended hours:** **MUST use STP LMT** (stop limit) when `stopExtendedHours=true`. Implementation: when extending hours, place STP LMT with limit-price = stop-price minus a slippage cushion (default 0.5% for liquid names, configurable). UI must clearly warn the user that extended-hours protection has fill-risk a regular STP doesn't, and may not fill on a hard gap.
4. **RTH default:** Daytime by default. **Per-position `stopExtendedHours: bool` field** required so Scott can opt in to extended-hours protection on specific positions. UI control in Command Center per position. IB API params on order: `outsideRth=true`, `orderType='STP LMT'`, `auxPrice=stopPrice`, `lmtPrice=stopPrice - slippage`.
5. **Audit retention:** Cold storage after 90 days. Implement via Atlas Online Archive when scale demands it; for v1 use a TTL index on outbox where status=DONE older than 90 days → archived collection (preserve forever, query rarely). FAILED commands NEVER auto-archive — they live in the hot collection until human review.

Every outbox enqueue checks `if (ownerId === DEMO_OWNER_ID) return;` BEFORE any IBKR write logic. - Rate limits hardcoded in bridge: max 5 orders/minute/symbol, max 50 orders/minute global.

## Critical "do not break" constraints

These are hard rules. Violating any of them risks money or credibility:

1. **Demo account writes are forbidden.** `ownerId === 'demo_fund'` short-circuits before any IBKR write. Hard-coded check, not a flag.
2. **Tightest stop always wins.** PNTHR never loosens a user-set stop. PNTHR adopts TWS values silently when TWS is tighter.
3. **Single canonical close path.** Both manual close (Command Center) and auto-close (Phase 2 IBKR exec) go through `recordExit()` in `exitService.js`. Adding a third write path is a regression.
4. **Refuse to auto-open without a stop.** Phase 3 returns early if `pnthrStop` is null. Never leave a position naked at the system level.
5. **OpEMA naming, never "21 EMA"** for stocks/sectors. Direction index (SPY/QQQ/MDY) is the ONLY 21W permitted; label it "21W Index EMA" so users never confuse the two.
6. **Master kill switch (`IBKR_WRITES_ENABLED=false`) overrides everything.** When false, Phase 4 logic is dead code. New session must verify this is OFF on every deploy.
